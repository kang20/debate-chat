# PR 코드 리뷰

PR을 분석하고 Pn 룰 기반으로 코드 리뷰를 수행합니다.

## 절차

1. `gh pr view $pr`로 PR 정보를 확인한다
2. `gh pr diff $pr`로 변경 내용을 분석한다
3. 아래 리뷰 관점에 따라 코드를 검토한다
4. Pn 룰에 따라 코멘트를 분류하여 리뷰 결과를 출력한다
5. 사용자가 요청하면 `gh pr review`로 GitHub에 리뷰를 제출한다

## 리뷰 관점

### 1. 아키텍처 일관성
- 프로젝트의 기존 패턴과 구조를 따르고 있는가
- 패키지 구조와 네이밍 컨벤션이 일관적인가

### 2. 대안 제시
- 더 나은 해결 방법이 있는가
- 사용된 라이브러리/패턴이 최선인가
- 성능, 유지보수 측면에서 더 적합한 접근법이 있는가

### 3. 버그 가능성
- NPE, 경계값, 동시성 이슈가 있는가
- 예외 처리가 적절한가
- 엣지 케이스를 고려했는가

### 4. 보안
- SQL Injection, XSS 등 OWASP Top 10 취약점이 있는가
- 민감 정보(비밀번호, API 키)가 노출되지 않는가
- 인증/인가 처리가 적절한가

### 5. 가독성
- 변수명, 메서드명이 의도를 명확히 전달하는가
- 불필요하게 복잡한 로직이 있는가
- 주석이 필요한 곳에 주석이 있는가 (과도한 주석은 아닌가)

### 6. 기술부채
- 임시 해결책(workaround)이 아닌 근본적인 해결인가
- 중복 코드가 발생하지 않는가
- 테스트가 충분한가

### 7. 순리
- "if문 하나만 넣으면 빠르게 해결할 수 있는데, 구조변경까지 해야 하나?"라는 고민이 드는 코드가 있는가
- 타협하지 않고 Best Practice를 따르고 있는가
- 단기적 편의보다 장기적 품질을 선택했는가

## Pn 룰

코멘트에 반드시 Pn 태그를 붙여 강조 정도를 표현한다.

### P1: 꼭 반영해주세요 (Request Changes)
- 서비스에 중대한 오류를 발생시킬 수 있는 경우
- 보안 취약점이 존재하는 경우
- 데이터 손실 가능성이 있는 경우

### P2: 적극적으로 고려해주세요 (Request Changes)
- 성능에 심각한 영향을 줄 수 있는 경우
- 아키텍처 일관성을 깨뜨리는 경우
- 수용하거나 적합한 의견으로 토론 권장

### P3: 웬만하면 반영해주세요 (Comment)
- 가독성을 크게 개선할 수 있는 경우
- 기술부채를 줄일 수 있는 경우
- 반영 불가 시 이유 설명 또는 후속 JIRA 티켓 생성 권장

### P4: 반영해도 좋고 넘어가도 좋습니다 (Approve)
- 더 나은 네이밍 제안
- 사소한 리팩토링 제안
- 무시해도 무방

### P5: 그냥 사소한 의견입니다 (Approve)
- 개인적 선호
- 참고용 정보 공유
- 무시해도 무방

## 출력 형식

```markdown
## PR 리뷰: #<pr-number> <pr-title>

### 요약
<PR의 목적과 변경 사항 요약>

### 리뷰 코멘트

#### P1 (꼭 반영)
- `파일명:라인` — 내용

#### P2 (적극 고려)
- `파일명:라인` — 내용

#### P3 (웬만하면 반영)
- `파일명:라인` — 내용

#### P4 (선택 반영)
- `파일명:라인` — 내용

#### P5 (참고)
- `파일명:라인` — 내용

### 총평
<전체적인 코드 품질 평가 및 승인/수정 요청 판단>
- P1 또는 P2가 있으면: Request Changes
- P3만 있으면: Comment
- P4, P5만 있거나 없으면: Approve
```

## 주의사항

- 인수 없이 호출하면 현재 브랜치의 PR을 자동으로 찾는다
- GitHub에 리뷰를 제출하기 전에 반드시 사용자에게 확인을 받는다
- 리뷰는 건설적이고 구체적으로 작성한다 — "이렇게 하면 안 됩니다"가 아니라 "이렇게 하면 더 좋겠습니다"
- 칭찬할 부분이 있으면 적극적으로 칭찬한다
